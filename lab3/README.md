# Лабораторная работа №3 — Менеджер виртуальной памяти

## Описание
Программа моделирует работу менеджера виртуальной памяти, преобразуя 16-битные логические адреса в физические с использованием TLB (Translation Lookaside Buffer), таблицы страниц и подкачки страниц по требованию из резервного хранилища (`BACKING_STORE.bin`). Приложение считывает список логических адресов из файла, выполняет трансляцию и выводит соответствующие физические адреса и значения байтов со знаком, хранящихся в физической памяти.

## Файлы проекта

| Файл | Назначение |
|------|------------|
| `vm_manager.c` | Исходный код программы |
| `addresses.txt` | Входной текстовый файл с 1000 логических адресов (каждый в новой строке) |
| `BACKING_STORE.bin` | Двоичный файл размером 65 536 байт, представляющий вторичную память (256 страниц по 256 байт) |
| `correct.txt` | Эталонный вывод для проверки корректности преобразований |
| `README.md` | Этот файл с описанием проекта |

## Архитектура виртуальной памяти
- **Размер виртуального адресного пространства**: 2¹⁶ = 65 536 байт
- **Размер страницы/фрейма**: 2⁸ = 256 байт
- **Количество страниц**: 256 (нумерация 0–255)
- **Количество фреймов**: 256 (при дополнительном задании может быть уменьшено до 128)
- **Формат логического адреса**: биты 15–8 — номер страницы, биты 7–0 — смещение

## Структуры данных в программе
- **TLB (Translation Lookaside Buffer)**: 16 записей, каждая хранит номер виртуальной страницы, соответствующий номер физического фрейма и бит достоверности. Замещение происходит по алгоритму FIFO.
- **Таблица страниц**: массив из 256 элементов. Для каждой виртуальной страницы хранится номер выделенного фрейма (или -1 при отсутствии) и флаг присутствия в памяти.
- **Физическая память**: массив `signed char` размером 65 536 байт. Адрес байта вычисляется как `frame * 256 + offset`.
- **Управление фреймами**: массив флагов занятости и указатель на следующий свободный фрейм (последовательное выделение).

## Алгоритм работы
1. Открыть файл `addresses.txt`, прочитать очередное целое число.
2. Ограничить значение младшими 16 битами (`addr & 0xFFFF`).
3. Извлечь номер страницы (`(addr >> 8) & 0xFF`) и смещение (`addr & 0xFF`).
4. Проверить TLB:
   - Если запись найдена (попадание) — взять номер фрейма из TLB, увеличить счетчик попаданий.
   - Если промах — перейти к таблице страниц.
5. Проверить таблицу страниц:
   - Если страница присутствует в памяти — взять номер фрейма, обновить TLB.
   - Иначе — зафиксировать страничную ошибку, увеличить счетчик ошибок.
6. Обработка страничной ошибки:
   - Выделить свободный фрейм (последовательный поиск первого незанятого).
   - Считать 256-байтную страницу из `BACKING_STORE.bin` (смещение в файле = `page_number * 256`).
   - Записать прочитанные данные в соответствующий фрейм физической памяти.
   - Обновить таблицу страниц (установить фрейм и валидность).
   - Добавить/обновить запись в TLB.
7. Вычислить физический адрес: `(frame << 8) | offset`.
8. Прочитать знаковый байт (`signed char`) из физической памяти по этому адресу.
9. Вывести результат: `Virtual address: <logical> Physical address: <physical> Value: <byte>`.
10. Повторять шаги 1–9 до конца файла.
11. Вывести статистику:
    - Частота попаданий в TLB (%)
    - Частота страничных ошибок (%)

## Компиляция и запуск

### Требования
- Компилятор GCC (или любой совместимый с C11)
- Наличие файлов `addresses.txt` и `BACKING_STORE.bin` в одной директории с исполняемым файлом

### Компиляция
```bash
gcc -std=c11 -Wall -o vm_manager vm_manager.c
